# Проектная Документация: "Список неотложных дел".

## 1. Введение

### 1.1 Цель проекта

Цель данного проекта заключается в создании приложения "Список неотложных дел", которое позволит пользователям управлять своими задачами, устанавливать приоритеты и эффективно планировать выполнение задач. Главной целью проекта является предоставление инструмента, способствующего повышению организованности и производительности пользователей.

### 1.2 Краткое описание проекта

Проект "Список неотложных дел" ориентирован на широкий круг пользователей, которые ищут простой и удобный способ управления своими задачами и обязанностями. Приложение предоставит функциональность для добавления, редактирования и удаления задач, а также установки приоритетов и уведомлений о ближайших сроках выполнения задач.

### 1.3 Обзор функциональных требований

Проект "Список неотложных дел" будет предоставлять следующие функциональные возможности:

- Добавление новых задач: Пользователи смогут добавлять задачи с указанием заголовка, описания и сроков выполнения.
- Установка приоритетов: Приложение позволит устанавливать приоритет задач для определения их важности.
- Редактирование задач: Пользователи смогут изменять заголовки, описания и сроки выполнения задач.
- Удаление задач: Возможность удалять задачи, которые более не актуальны.
- Уведомления: Приложение будет отправлять уведомления и напоминания о приближающихся сроках выполнения задач.


## 2. Требования

### 2.1 Функциональные требования

Функциональные требования определяют, какие функции и возможности должны быть реализованы в проекте "Список неотложных дел". Ниже приведен список основных функциональных требований:

- **Добавление задач**:
  - Пользователи могут добавлять новые задачи, указывая заголовок, описание и сроки выполнения.

- **Установка приоритетов**:
  - Приложение должно позволить пользователям устанавливать приоритеты для задач (например, высокий, средний, низкий).

- **Редактирование задач**:
  - Пользователи могут вносить изменения в существующие задачи, включая заголовок, описание и сроки выполнения.

- **Удаление задач**:
  - Пользователи должны иметь возможность удалять задачи, которые больше не актуальны или завершены.

- **Уведомления и напоминания**:
  - Приложение должно отправлять уведомления и напоминания пользователю о приближающихся сроках выполнения задач.

### 2.2 Нефункциональные требования

Нефункциональные требования определяют аспекты системы, не связанные напрямую с функциональностью, но важные для качества и производительности приложения. Ниже приведены некоторые нефункциональные требования:

- **Производительность**:
  - Приложение должно обеспечивать отзывчивость и быстродействие, даже при большом количестве задач в списке.

- **Безопасность**:
  - Данные пользователей должны храниться в безопасности, и доступ к ним должен быть ограничен.

- **Кросс-платформенность**:
  - Приложение должно быть доступным на различных платформах, включая мобильные устройства (iOS и Android) и веб-браузеры.

### 2.3 Требования к интерфейсу

Требования к интерфейсу определяют структуру и дизайн приложения:

- **Пользовательский интерфейс**:
  - Интерфейс должен быть интуитивно понятным и простым в использовании.
  
- **Дизайн**:
  - Дизайн приложения должен быть современным и привлекательным для пользователей.

- **Макеты и прототипы**:
  - Перед разработкой приложения должны быть созданы макеты пользовательского интерфейса и прототипы для оценки дизайна и взаимодействия.

    
## 3. Архитектура

### 3.1 Общая архитектура приложения

Общая архитектура приложения "Список неотложных дел" будет основана на клиент-серверной модели. Приложение будет состоять из двух основных компонентов: клиентской части, доступной на мобильных устройствах и веб-браузерах, и серверной части, которая будет отвечать за обработку запросов и управление данными.

- **Клиентская часть**: Клиентская часть будет разработана с использованием современных мобильных фреймворков для iOS и Android, а также веб-технологий для поддержки веб-браузеров. Она обеспечит пользователей доступом к функциональности приложения и взаимодействием с сервером.

- **Серверная часть**: Серверная часть приложения будет построена на основе серверных технологий, таких как Node.js или Python, и будет взаимодействовать с базой данных для хранения информации о задачах и пользователях. Она также будет отвечать за отправку уведомлений и обработку запросов от клиентов.

### 3.2 Описание компонентов

Основные компоненты приложения включают:

- **Интерфейс пользователя (UI)**: Компонент, отвечающий за отображение информации пользователю и взаимодействие с ним через графический интерфейс. Включает в себя экраны задач, формы ввода, и интерфейс для установки приоритетов и редактирования задач.

- **Управление задачами**: Компонент, отвечающий за создание, редактирование, и удаление задач. Включает в себя бизнес-логику для управления задачами и их приоритетами.

- **Серверная логика**: Компонент, работающий на сервере, который обрабатывает запросы от клиентов, управляет данными, и отправляет уведомления.

- **База данных**: Компонент, отвечающий за хранение информации о задачах и пользователях.

### 3.3 Технологии и стек

Технологический стек для разработки приложения включает следующие элементы:

- **Мобильная разработка**: Для разработки мобильной части приложения будут использованы средства и фреймворки, такие как Swift и SwiftUI для iOS, Kotlin и Android Studio для Android.

- **Веб-разработка**: Для веб-версии приложения будут использованы HTML, CSS и JavaScript. Возможно использование фреймворков, таких как React или Angular.

- **Серверное программирование**: Для серверной части приложения будут выбраны технологии, которые обеспечивают высокую производительность, такие как Node.js или Python с использованием фреймворков Express.js или FastAPI.

- **База данных**: Для хранения данных будет использоваться реляционная или NoSQL база данных, такая как PostgreSQL, MongoDB или Firebase Realtime Database.
